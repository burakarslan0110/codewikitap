## Why you would use this

A familiar pattern when working with coding agents like Claude Code, Cursor, Codex CLI, or Gemini CLI: you ask the agent how Next.js's App Router cache behaves, and it answers from training data — possibly correct, possibly outdated, possibly hallucinated. To be sure, you open the Next.js documentation yourself.

**CodeWiKiTap closes that gap.** For every direct dependency in your project, it makes the technical documentation hosted on [Google CodeWiki](https://codewiki.google) — re-generated by Gemini on every PR merge, pinned to a specific commit SHA, validated against the repository — available to the agent as structured context. The result is **answers that cite, link, and pin to a source** rather than answers that approximate.

### Concrete benefits

| Before | After |
|---|---|
| The agent guesses; sometimes the answer is outdated or simply wrong | The agent reads from the official documentation and cites it |
| Suggestions come from old Stack Overflow patterns in the training data | The agent consults documentation that was regenerated on the latest PR merge |
| You have to tell the agent which library has docs and which doesn't | At session start, the agent already has a coverage report for every direct dependency |
| You manually find the Next.js / React / Prisma page and paste it in | The agent navigates straight to the relevant page or sub-section in a single tool call |

One-line install (`npx codewikitap`). No API keys, no cloud dependencies, no telemetry. Everything runs on your local machine.

### How it differs from context7

[context7](https://github.com/upstash/context7) serves pre-indexed documentation for a broad set of popular libraries. CodeWiKiTap draws on a different source: **Google CodeWiki**, a Gemini-synthesized documentation layer that produces deeply repo-specific content for every public GitHub repository — not just API references, but architecture diagrams, module relationships, data flows, and file-level explanations. This content is re-produced on every PR merge, so version drift is structurally avoided.

Practical differences:

- **Context depth.** CodeWiki content goes beyond API surface and into architectural and inter-module reasoning, including Mermaid-style diagrams.
- **Project-awareness.** context7 is invoked on demand (you or the agent name the library). CodeWiKiTap scans the project's manifest at session start and has a coverage report ready before the first question.
- **Knowledge graph.** CodeWiKiTap extracts typed relationships from CodeWiki pages and exposes them for traversal (see below); context7 has no equivalent.
- **Coverage.** CodeWiki currently supports only public GitHub repositories — context7's broader library index serves a different use case. The two are not mutually exclusive and can coexist in the same agent configuration.

### Knowledge graph over the documentation

While indexing CodeWiki content, CodeWiKiTap extracts a **typed knowledge graph** in the same pass. Five stored edge kinds:

- `code_ref` — a documentation page references a specific source file
- `diagram_edge` — an edge inside a Mermaid / architecture diagram
- `diagram_member` — a node belongs to a diagram cluster
- `section_link` — anchor link between sections of the same page
- `cross_repo_ref` — a doc references a different upstream repository

A query-time `dep_link` is derived from project scan + wiki index status. The `find_neighbors` tool then lets the agent ask, for example:

- "Which documentation pages reference `src/auth/login.ts`?"
- "What is connected to the `AuthRouter` node in the auth diagram?"
- "Where do this repo's docs cite `vercel/next.js`?"

The graph is produced as a byproduct of chunk indexing — there is no separate model download, no separate index file, no extra disk cost. An optional `query` parameter on `find_neighbors` re-ranks typed neighbors by semantic similarity using the existing embedder, lazy-loaded only when supplied.

## How it works in practice — a walkthrough

Suppose your Next.js 14 App Router project has a caching problem: a `revalidatePath` call isn't clearing the cache after a mutation. You ask the agent:

> "Why isn't my `revalidatePath` call refreshing the cached fetch in this component?"

**Without CodeWiKiTap:** The agent typically lists a few possibilities from general patterns it remembers — "maybe the cache key doesn't match", "maybe tagging is missing". To get a definitive answer, you would open the Next.js documentation yourself.

**With CodeWiKiTap:** The agent runs four steps in the background.

1. **Project orientation.** `list_project_dependencies` was already called at session start. The agent knows `next` resolves to `vercel/next.js` and that the repo has 18 pages of CodeWiki documentation.

2. **Hybrid retrieval.** A `find_chunks` query — `"revalidatePath cached fetch server component"` — is scoped to `vercel/next.js`. The retriever runs BM25 (keyword) and dense-vector (semantic) searches in parallel, fuses the two ranked lists via Reciprocal Rank Fusion, then feeds the top candidates into a cross-encoder reranker. Returned chunks expose `vectorScore`, `bm25Score`, `rrfScore`, and `rerankScore` — the ranking is auditable.

3. **Targeted detail.** The top result is App Router → Caching → On-demand revalidation. The agent calls `get_page` on that subsection and receives the full Markdown — diagrams, code samples, and the `revalidatePath(path, type)` signature included.

4. **Knowledge-graph traversal (optional).** To see architectural relationships, the agent calls `find_neighbors` for the `RouteCache` node — returning `FullRouteCache`, `DataCache`, `RouterCache`, `RequestMemoization`, each with its own source citation.

The agent's response then reads roughly like this:

> "According to the Next.js documentation ([source pinned to commit `a1b2c3d`](https://codewiki.google/...)), `revalidatePath(path)` without a type argument invalidates only the **route segment** cache. Your `force-cache` fetch result lives in the **Data Cache**, which is keyed independently. To invalidate both, use `revalidatePath(path, 'page')`; alternatively, tag the fetch and call `revalidateTag(tag)`."

The answer is grounded in current documentation, includes a clickable citation, and is verifiable by the reader.

## Installation

One line:

```bash
npx codewikitap
```

On first run, Playwright's `chromium-headless-shell` (~30 MB) is downloaded. This is required because CodeWiki is an Angular SPA and a plain HTTP request returns an empty shell. After the one-time download, startup is instant.

You do not invoke `codewikitap` directly; your coding agent launches it as a child process. Add the four-line configuration block below for your agent of choice.

### Claude Code

`~/.claude/mcp.json` (user-level) or `.mcp.json` at the project root:

```json
{
  "mcpServers": {
    "codewikitap": {
      "command": "npx",
      "args": ["-y", "codewikitap"]
    }
  }
}
```

Or install via the Claude plugin marketplace (one command):

```text
/plugin marketplace add burakarslan0110/codewikitap
/plugin install codewikitap@burakarslan0110-codewikitap
```

### Cursor

`~/.cursor/mcp.json` (or `<project>/.cursor/mcp.json`) — same JSON shape as Claude Code.

### Codex CLI

`~/.codex/config.toml`:

```toml
[mcp_servers.codewikitap]
command = "npx"
args = ["-y", "codewikitap"]
```

### Gemini CLI

`~/.gemini/settings.json` under `mcpServers` — identical shape to Claude Code.

### Qwen Code, opencode, Antigravity

All accept an `mcpServers` (or `mcp`) JSON object — the same shape as Cursor / Claude Code. Paste into the agent's config file, restart, done.

## What the agent gains — 7 tools

You do not call these directly; the agent does. The capability list is worth knowing:

| Tool | What it is for |
|---|---|
| `list_project_dependencies` | Called once at session start. "Which dependencies are in this project and which have CodeWiki documentation?" — the foundation of the entire session. |
| `resolve_repo` | Resolves a vague name like "react" to `facebook/react`. Skipped when owner/repo is already known. |
| `list_pages` | The table of contents for a library — what topics and section headings exist. |
| `get_page` | Fetches a page (or sub-section) as full Markdown, including diagrams, code samples, and the citation footer. |
| `find_chunks` | Hybrid retrieval: BM25 + vector + Reciprocal Rank Fusion + cross-encoder reranking. Returns the most relevant paragraphs along with five separate scores for ranking transparency. |
| `find_neighbors` | Knowledge-graph traversal. "Which docs reference this file?", "What is connected to this diagram node?", "Where does this doc cite another repo?" |
| `request_indexing` | Pre-warming. Indexes a repo ahead of subsequent queries so the first `find_chunks` call returns immediately rather than racing the indexer. |

The seven names are **locked**. No plugin or hook can quietly add an eighth tool — the server rejects it at the code level.

## Supported project types

CodeWiKiTap reads your manifest to learn what your project depends on. Eleven ecosystems are supported out of the box, with workspace traversal where applicable:

- **JavaScript / TypeScript:** `package.json` + npm/pnpm/yarn `workspaces`, `pnpm-workspace.yaml`
- **Python:** `requirements.txt`, `pyproject.toml` (PEP 621 + Poetry)
- **Go:** `go.mod`, `go.work` (workspace-aware)
- **Rust:** `Cargo.toml` + `[workspace] members` (literal and glob)
- **PHP:** `composer.json` (platform packages filtered automatically)
- **Java:** `pom.xml` — full property resolution + BOM imports (recursive, cycle-safe) + `<parent>` POM resolution + aggregator-pom `<modules>` traversal
- **Gradle:** `gradle/libs.versions.toml` Version Catalogs + `settings.gradle(.kts)` subproject discovery + per-subproject `build.gradle(.kts)` parsing
- **Ruby:** `Gemfile.lock` (preferred) + `Gemfile` regex fallback
- **.NET:** `*.csproj` + `Directory.Packages.props` (Central Package Management) + `*.sln`-driven discovery

Saying "react" is sufficient — the resolver maps it to `facebook/react`. Maven Central, RubyGems, NuGet, crates.io, Packagist, and the npm registry are all integrated.

## What this is not

- **Not an AI model.** No model is bundled; CodeWiKiTap improves the *context quality* delivered to the AI the agent already uses.
- **Not a documentation generator.** Content is not produced here. CodeWiKiTap fetches what Google CodeWiki has already produced with Gemini.
- **Not a cloud service.** Nothing leaves the local machine. Calls happen locally, the cache is local (`~/.cache/...`), telemetry is zero.
- **Public repositories only.** Google CodeWiki currently supports only public GitHub repositories; private-repo access is a waitlisted Gemini CLI extension.

## Configuration

Defaults are sufficient for most cases. When fine-tuning is needed, the most commonly used variables:

| Variable | Default | Purpose |
|---|---|---|
| `LOG_LEVEL` | `info` | Stderr log verbosity (`debug` / `info` / `warn` / `error`). |
| `CODEWIKI_INCLUDE_DEV_DEPS` | off | When on, `devDependencies` are also scanned (useful when test-tool documentation matters). |
| `CODEWIKI_DISABLE_WATCH` | off | When on, manifest changes are not watched (useful in CI/CD). |
| `CODEWIKI_DISABLE_KG` | off | When on, the knowledge graph is not built; `find_neighbors` is disabled. |
| `CODEWIKI_DISABLE_PREWARM` | off | When on, startup auto-prewarm is skipped. |
| `CODEWIKI_FORCE_NO_BM25` | off | When on, `find_chunks` runs in vector-only mode (the BM25 branch is skipped). |
| `CODEWIKI_RERANK_TOP_N` | `50` | Candidate count passed to the reranker. |

Full reference is in [CONTRIBUTING.md](CONTRIBUTING.md).

## Things worth knowing

- **AI-generated content.** Google CodeWiki pages are produced by Gemini and may contain errors. CodeWiKiTap appends a byte-equal citation footer to every chunk and every page response — the source link is meant to be used for verification.
- **Rate limit.** Because CodeWiki is an Angular SPA with active bot detection, CodeWiKiTap self-throttles to one page load per four seconds per origin. Typical interactive use does not feel this; bulk-indexing many repos at once will.
- **Install footprint.** First run downloads ~150 MB of Chromium (Playwright headless). The first `find_chunks` call downloads ~50 MB of ONNX models (embedding + reranker). Both are one-time, persistently cached.
- **Scope.** Only direct dependencies are scanned. `peerDependencies` and transitive deps are currently excluded; npm `optionalDependencies` are included by default.
